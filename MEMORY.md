## Drawing Analyzer — Key Learnings
- Pipeline order: viable area → text removal → hairlines → pipes → symbols (strip easy stuff first)
- **XY axis rule**: pipes are ALWAYS horizontal or vertical on plans. Anything diagonal = architectural interference
- Plans are rotated so pipe runs align to H/V axes
- Each engineering firm has different symbol styles — no industry standard
- Library approach: user screenshots/crops a symbol → adds to search library → system finds matches
- Symbol detection endgame: strip everything that isn't a symbol, cluster what's left, count per cluster
- Tremont St plan: 63 heads (confirmed by Thomas manual count)
- Lane Park plan: 5 heads (I miscounted as 4 — be careful on small plans)
- Testing files in iCloud: ~/Library/Mobile Documents/com~apple~CloudDocs/Testing Files/
- **Symbol size consistency**: All symbols on floor plans are the same pixel size (generated by AutoCAD/Bluebeam). Detail cutaway sections are vector-scaled zooms, so symbols in details are LARGER. Use size as a filter to avoid double-counting from detail views.
- **Approach pivot**: Vision AI analysis beat the pixel pipeline on first try (63/63 on Tremont). Skip complex section detection / layer stripping — just render + vision + confirm.
- **Current approach (2026-03-23)**: Tile-based flood fill for section detection (all local CV2, zero API cost). PDF text + diagonal line stripping before detection. Two-step: detect border → refine within. 90° polygon overlay. Manual "Scan for Items" button to control API spend.
- **COST ALERT**: Opus conversations are extremely expensive ($67 in one session). ALWAYS use Sonnet for coding work. Only use AI counting as last resort — prefer template matching library (zero cost).
- **Next step**: Build symbol template matching library so counting is 100% local too. No AI API calls in the pipeline.
